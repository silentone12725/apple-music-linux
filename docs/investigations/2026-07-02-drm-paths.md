# DRM Path Investigation — 2026-07-02

## Motivation

After the `engine/apple` provider was wired up, `POST /api/v1/playback` failed with:

```
open audio stream: media playlist has no encryption info
```

The question was: is this a bug in the HLS parser or in the URL source selection?

## Method

Used `cmd/hlscompare` to fetch media playlists from both candidate sources
and print their raw `EXT-X-KEY` URIs.

```
go run ./cmd/hlscompare 1488408568 us
```

`cmd/hlscompare` auto-fetches the bearer token from the Apple Music web app.
MUT read from `config.yaml`.

## Conclusions

**Fact:** The two paths serve different files from different CDN domains with
different EXT-X-KEY formats:
- `aod-ssl` / `28:ctrp256` → `data:;base64,[kid]` — CTR, Widevine-compatible
- `aod` / catalog enhanced-HLS → `skd://itunes.apple.com/…` — CBCS, FairPlay

**Fact:** `strings.SplitN("skd://itunes.apple.com/…", ",", 2)` returns one part,
so `engine/hls.OpenMedia` correctly leaves `EncryptionInfo = nil`.

**Conclusion (supported by evidence):** The regression in the new engine was caused
by selecting the catalog enhanced-HLS path for AAC, which exposes FairPlay CBCS
keys the Widevine license provider cannot use. Switching to the webplayback
`28:ctrp256` asset gives a CTR-encrypted playlist the engine can handle.

**Open question:** Whether ALAC/Atmos content also has CTR-compatible webplayback
assets (and which flavor those would be) is not yet investigated.

## How to rerun

```bash
go run ./cmd/hlscompare <adamID> [storefront]
```

Requires `config.yaml` with a valid `media-user-token`. Bearer token is
auto-fetched from the Apple Music web app if not set or stale.
