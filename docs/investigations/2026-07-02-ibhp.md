# IBHP Asset Variant Investigation — 2026-07-02

## What is ibhp?

The webplayback API returns six asset flavors for adamID 1488408568 (and likely
for all tracks). Two of those flavors use a key delivery format (`KEYFORMAT=identity`)
that is distinct from both the CTR/Widevine and CBCS/FairPlay paths:

| Flavor      | Description |
|-------------|-------------|
| `37:ibhp256` | 256 kbps AAC, identity-based HLS playback |
| `38:ibhp64`  | 64 kbps AAC, identity-based HLS playback |

"ibhp" appears to stand for **identity-based HLS playback** — HLS standard
AES encryption where the key is fetched directly via HTTPS rather than through
a DRM system.

## DRM format analysis

`KEYFORMAT=identity` is the HLS standard for AES-128 CBC stream encryption.
The HLS player is expected to:
1. HTTP GET the key URI (with appropriate authentication headers, presumably
   cookies or an `Authorization` header).
2. Receive a raw 16-byte AES-128 key in the response body.
3. Decrypt each MPEG-TS or fMP4 segment using `AES-128-CBC` with the
   explicit IV from the playlist.

This is **not** FairPlay Streaming and **not** CENC/Widevine. It predates both
and is the original HLS content protection mechanism (RFC 8216 §4.3.2.4).

Key observations:
- The key URI hostname is `play.itunes.apple.com` — Apple's playback server,
  the same host as the FPS endpoint used for CBCS.
- The content identifier in the URI (`afs_1488408568_31_a_`) is the same
  identifier used in the cbcp256 `skd://` URI. The 256 kbps ibhp and cbcp
  variants share the same content identifier and the same init segment file.
- The IV is explicit and differs between ibhp256 and ibhp64 (different keys,
  different IVs, same track at different bitrates).
- 21 segments (vs 14 for catalog enhanced-HLS) — WebPlayback uses a
  different segmentation than the catalog path.

ibhp and cbcp share init segments, suggesting the encrypted segment data
may also be the same file encrypted with different keys. This would be
consistent with Apple delivering the same audio content through both FairPlay
and AES-identity paths for different client capabilities.

## Wrapper support

The wrapper server (`wrapper-rootless`) exposes:
- HTTP endpoint for Widevine CTR key delivery (used by `runv3`)
- TCP socket for FairPlay CBCS decryption proxy (used by `runv2`)

**The wrapper has no path for ibhp/identity.** Key delivery requires a direct
HTTPS GET to `play.itunes.apple.com/WebObjects/MZPlay.woa/hls/identityKeyRequest/…`
with authenticated Apple Music session credentials. This call goes directly to
Apple's servers without wrapper involvement.

## Why ibhp is not planned

Given the wrapper constraint (no direct Apple server calls), implementing ibhp
would require:
1. Passing the user's session credentials through the engine to make direct
   Apple API calls.
2. Or adding a new ibhp proxy path to the wrapper.

Neither is warranted given that cbcp256 and ctrp256 already cover 256 kbps AAC
for the same content. ibhp appears to be a fallback for clients that cannot
handle FairPlay or Widevine.

## Open questions

1. **What authentication does the identityKeyRequest endpoint require?**
   Likely the Apple Music `media-user-token` header (same as webplayback). Not
   verified — no request has been made to this endpoint.

2. **Is ibhp a legacy format or actively maintained?** The cbcp variant covers
   the same content. ibhp may exist for older Apple TV or Safari HLS clients
   that predate FairPlay Streaming.

3. **Do all tracks expose ibhp?** Only verified for adamID 1488408568 (Blinding
   Lights). Tracks without `lossless` or `atmos` AudioTraits likely also have
   ibhp variants; not checked.

4. **Segment encryption scheme:** SAMPLE-AES with identity KEYFORMAT can mean
   different things. The segments may be MPEG-TS with AES-128-CBC, or fMP4 with
   SAMPLE-AES (CBC at the sample level, not the segment level). Not verified —
   would require fetching and inspecting a segment.

## Conclusion

ibhp is understood well enough to explain why it exists and why it is not
implemented. It does not represent a content type unavailable through ctrc or
cbcs paths. No further investigation is planned unless a content type is found
that is only accessible via ibhp.
